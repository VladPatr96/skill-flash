# /memo — служебная записка в issue

`/memo <задача>` превращает сырую формулировку в самодостаточную спеку issue. Это вход производственного конвейера office; исполнитель получает только указатель на issue и не должен исследовать задачу заново.

## Протокол

1. **Сигнал.** Классифицируй вход: feature / bug / chore / tooling / ritual. Явно отдели факт от гипотезы.
2. **Readiness.** Если задача вне скоупа проекта или требует решения фаундера, не диспатчь: создай issue с `status:blocked` или перенеси в `.out-of-scope/` с причиной. Готовая задача получает `status:spec-ready`.
3. **Complexity-gate.** Оцени S/M/L:
   - S: один файл или один артефакт, понятная проверка;
   - M: несколько файлов, но без архитектурной развилки;
   - L: эпик или неизвестная архитектура, сначала kickoff/декомпозиция.
4. **Conflict metadata.** Запиши в спеку `conflict_paths`, `parallel_safe: yes/no`, `depends_on`, чтобы безопасно параллелить непересекающиеся задачи.
5. **Инструмент.** Ответь в спеке: есть ли повторяемое ядро? Если да и вынос стоит ≤20% задачи, добавь шаг создать `company/<отдел>/tools/`; если дороже — отдельный issue `type:tooling`.
6. **Спека.** Используй дисциплину `writing-specs` и шаблон `skills/office/templates/spec-issue.md`. Тело issue переписывается целиком.

## Выход

- GitHub issue с полной спекой, лейблами `type:*`, `dept:*`, `status:spec-ready|blocked`.
- Короткий ответ пользователю: `завёл #N, статус <status>, в конвейере после #M`.

## DoD

- [ ] Исполнитель может выполнить issue без разведки и вопросов.
- [ ] В спеке есть complexity, conflict metadata и ответ про инструмент.
- [ ] DoD содержит конкретную команду/сценарий и требование приложить свежий вывод проверки.
