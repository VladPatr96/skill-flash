# Blueprint: Агентство услуг в соло

**Кому подходит:** соло-специалист или маленькое агентство на заказных проектах (разработка, дизайн, маркетинг, консалтинг), где фаундер — единственная точка контакта с клиентами, а исполнение масштабируется командой AI-агентов.

В отличие от продуктовых blueprint'ов здесь нет одного продукта: есть портфель клиентских проектов, и unit-экономика (часы + токены vs оплата) считается по каждому отдельно.

## Функции бизнеса

лидогенерация → бриф, оценка, КП → деливери проекта → приёмка (QA по брифу) → сдача клиенту → account и финансы (маржа) → engineering конвейера

## Оргпроект (профильные отделы к ядру board/engineering/qa)

Мультипроектность — основа оргпроекта: клиент = отдельный milestone (крупный или долгий клиент — отдельный репозиторий); state.md компании ведётся по клиентам, не одним общим файлом: `company/clients/<client>/state.md`.

Вход нового клиента стандартизирован: bootstrap клиентской папки/репо → назначение delivery-lead → brief.md — до старта любых работ по проекту.

| Отдел | Лейбл | Миссия | Агенты (модель) | Скиллы из реестра |
|-------|-------|--------|-----------------|-------------------|
| sales (pre-sale) | dept:sales | снять бриф структурированным интервью (паттерн grill-me — дожимать каждую ветку до ясности), оценить объём и риски, упаковать в КП | closer/интервьюер (Sonnet), estimator (Sonnet) | pm-prd (бриф → структура), pm-prioritize (объём КП/пакета) |
| delivery | dept:delivery | исполнить проект по брифу силами ядра engineering или design (по профилю заказа); сложные развилки — на ceo-council | delivery-lead (Sonnet) + агенты ядра по профилю (Sonnet/Haiku) | ceo-council (сложные решения); art-director, design-minimal, html-draft (профиль дизайн); codex plugin (профиль разработка, второе мнение) |
| qa-client (приёмка) | dept:qa-client | сверить сделанное с брифом и КП до показа клиенту — отдельный от инженерного QA финальный фильтр | acceptance-checker (Sonnet) | code-review, verify (профиль разработка); чек-лист брифа (остальные профили) |
| account | dept:account | вести статус-коммуникацию с каждым клиентом: апдейты и отчёты — черновиками, без прямой отправки | account-writer (Sonnet) | manager (статус трека по клиенту), gh-issues (сводка по проекту) |
| finance | dept:finance | учитывать часы фаундера и токены агентов по клиенту/проекту, считать unit-экономику и маржу | finance-tracker (Haiku) | cc-analytics (токены/использование), pm-metrics (тренды маржи) |

Лейбл dept:qa-client намеренно отделён от ядрового dept:qa: инженерное/дизайн-качество проверяет ядро внутри delivery, а qa-client — последний фильтр «соответствует ли брифу» перед клиентом.

Ownership: sales — `company/clients/<client>/brief.md` и КП; delivery — рабочие папки/репо проекта; qa-client — чек-лист приёмки против брифа; account — черновики писем/отчётов (без отправки); finance — `company/clients/<client>/finance.md`; ядро engineering/design — код и артефакты по профилю.

**Критично для агентства вдвойне:** любая коммуникация с клиентом и любая сдача работ — только через фаундера, без исключений. Ни сообщение, ни файл, ни демо не уходят клиенту напрямую от агента: на кону не просто качество, а деньги и репутация.

## OKR-заготовки

| KR | Метрика | Как меряем |
|----|---------|-----------|
| Загрузка | проектов в работе одновременно vs лимит потока | account weekly, `company/clients/` |
| Маржа | (оплата − часы фаундера×ставка − стоимость токенов агентов) / оплата, по каждому проекту — токены считать обязательно, не только часы человека | finance-лог при закрытии проекта |
| Скорость цикла | медиана «бриф → сдача» по типу проекта | sales+delivery лог |
| Повторные клиенты | % клиентов со 2+ проектом за период | finance/account квартально |

## Гейты этапов (роадмап)

0. Первый проект по конвейеру (весь путь бриф→сдача проходит через отделы, вручную сверено фаундером) → 1. Поток (2-3 параллельных проекта без потери маржи и качества) → 2. Продуктизация услуг (типовые пакеты и прайс вместо разовых КП) → 3. Масштабирование (найм людей/партнёров, новые направления и каналы лидов).

Сигнал перехода 0→1: первый проект сдан без овербюджета по токенам и принят клиентом с первого показа. Сигнал 1→2: 3+ похожих проекта подряд — пора паковать в фиксированный пакет и прайс.

## Типовые первые задачи

- `type:feature` шаблон брифа + grill-me интервью для {профиль услуги} (sales)
- `type:feature` bootstrap клиентской папки/репо + назначение delivery-lead при новом контракте (delivery)
- `type:feature` чек-лист приёмки qa-client по брифу перед показом клиенту (qa-client)
- `type:tooling` учёт часов и токенов по клиенту в `company/clients/<client>/finance.md` (finance)
- `type:chore` шаблон статус-письма клиенту + правило review-gate (account)

## Промт-пак отдела (слот-формат; копируется в company/<отдел>/prompts.md при найме)

Промты sales/account, чей результат видит клиент, всегда идут через review-gate фаундера — отмечено в теле промта.

**Интервью-бриф клиента** *(sales)*
```
проведи структурированное интервью по заявке {client}: цель проекта, аудитория, бюджет, срок, критерии успеха, что уже пробовали, ограничения.
Паттерн grill-me — дожимай каждую ветку до ясности, не переходи дальше, пока вопрос не закрыт. Зафиксируй в company/clients/{client}/brief.md
```
> профиль услуги: разработка / дизайн / маркетинг / консалтинг — определяет набор уточняющих вопросов

**Оценка проекта** *(sales)*
```
по brief.md {client} сделай декомпозицию на этапы и задачи, оцени часы фаундера и токены агентов по каждому, топ-3 риска.
Unit-оценка: цена ≥ (часы×ставка + токены×buffer) / целевая маржа {%}
```
> целевая маржа: 40-60% · buffer на токены: +20% к расходу похожего прошлого проекта

**КП по брифу** *(sales)*
```
собери КП для {client} из brief.md и оценки: объём по этапам, срок, цена, что входит/не входит, 2 пакета (базовый/расширенный).
Черновик — на review-gate фаундеру, клиенту не отправлять
```

**Статус-письмо клиенту** *(account)*
```
подготовь статус-письмо {client} за {период}: что сделано, что в работе, блокеры (если есть), следующий шаг и дата.
REVIEW-GATE: черновик для фаундера; отправка клиенту — только после его правки и явного одобрения
```

**Пост-мортем проекта с маржой** *(finance)*
```
по завершённому {project}: план vs факт часов фаундера, токены агентов по этапам, итоговая маржа = (оплата − часы×ставка − токены×цена) / оплата.
Выведи: 1-3 причины отклонения от плана, поправку для оценки следующего похожего проекта
```

## Отчёты

Weekly (общий шаблон) + от account: статус по каждому активному клиенту; от finance: маржа по проекту при закрытии, сводная маржа и загрузка за неделю. Месячный business review — обязателен с этапа 1: загрузка, маржа, повторные клиенты, конверсия бриф→КП→сделка.
