## Orchestration workflow (Office company mode)

Оркестратор (сессионная модель) — CEO: планирует, пишет спеки, резолвит развилки, синтезирует. Держит свой контекст пустым — читает и делает команда:

- Глубокий анализ (архитектура, сложный дебаг, алгоритмы) → `deep-reasoner` (Opus)
- Механика по готовой спеке (код, тесты, правки) → `fast-worker` / агент отдела (Sonnet)
- Чтение и инвентаризация → Haiku-скауты
- Приёмка → `verifier` (Sonnet, свежий контекст) исполняет проверку из DoD
- Свежий взгляд → Codex `/codex:rescue --background` — пир, не ревьюер; high-stakes решения — deep-reasoner + Codex параллельно на одной задаче, не показывая ответы друг другу

Правила конвейера: issue-first (любая задача сначала issue со спекой в теле), диспатч указателем на issue, очерёдность по пересечению файлов, закрытие только после verifier.

**Документация живая**: README, цели и `company/state.md` обновляются в момент изменения; приёмка задачи, менявшей поведение, включает обновление затронутых доков. Новая сессия начинается с чтения `company/state.md` — не с раскопок.

**Простота побеждает**: сначала самое простое и дешёвое работающее решение, доработка — отдельным issue по фактам использования. **Правило трёх**: рутина повторилась ≥3 раз → issue `type:tooling` — скрипт в `company/<отдел>/tools/` или скилл для агентов.

**Query-first**: контекст добывается сначала `graphify query "вопрос"` (карта в `graphify-out/`), чтение файлов — точечно по результатам подграфа. Grep по всей базе — крайняя мера.

Оргструктура, чартеры отделов и ownership-карта: `company/README.md`. Агент меняет только пути из ownership-карты своего отдела.

## Правила кода

1. **Дробление.** Файл — не больше ~1000 строк. Код делится на маленькие модули с **явными связями**: импорты и контракты видны, никакой магии. Маленький кусок проще править, дешевле читать агенту и безопаснее верифицировать.
2. **Оценка влияния.** Перед любой правкой выясняется, кто зависит от изменяемого (`graphify query "what depends on X"` или карта кода); влияние фиксируется в спеке issue. Правка без оценки влияния не диспатчится.
3. **Только точечные правки.** Файл никогда не переписывается целиком — только точечные edit'ы. Полная перезапись существующего файла — красный флаг на верификации.
4. **Горячие части — выносить.** Если один и тот же кусок файла меняется почти в каждой задаче — он выносится в отдельный файл (issue `type:chore`), чтобы правки стали локальными.
5. **Комментарии на русском.** У каждой функции/модуля — комментарий: что делает, как работает, откуда берутся данные (входы, источники, побочные эффекты). Код и идентификаторы — на английском, пояснения — на русском.
6. **Тесты — до кода.** Сначала тест, фиксирующий требование или воспроизводящий баг (red), потом код до зелёного (green). Спека без проверяемого DoD не диспатчится; фича без теста не принимается верификатором.

## Behavioral guidelines

Осторожность важнее скорости; для тривиальных задач — здравый смысл.

1. **Think Before Coding** — предположения проговаривать явно; несколько трактовок — показать, а не выбрать молча; есть путь проще — сказать; непонятно — остановиться и назвать, что именно.
2. **Simplicity First** — минимум кода, решающего задачу; без спекулятивных фич, абстракций на одноразовый код и обработки невозможных сценариев. «Сеньор назвал бы это переусложнённым?» — упрощай. Простое и дешёвое сейчас, идеальное — потом отдельной задачей, если понадобится.
3. **Surgical Changes** — трогать только необходимое; не улучшать соседний код и не рефакторить работающее; стиль — как вокруг; удалять только сирот от собственных правок; каждая изменённая строка ведёт к запросу.
4. **Goal-Driven Execution** — задачу превращать в проверяемую цель («почини баг» → «тест, воспроизводящий баг, затем зелёный»); многошаговое — план `[шаг] → проверка: [чек]`; крутиться до верификации.
