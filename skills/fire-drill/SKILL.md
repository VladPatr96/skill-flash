# /fire-drill — протокол инцидента

Цель — восстановить управляемость: сначала сигнал и доказательства, потом гипотезы, только затем изменение. Рестарт, деплой, удаление данных и внешние действия требуют greenlight.

## Фазы

1. **Сигнал.** Зафиксируй симптом, время, затронутых пользователей, что изменилось недавно. Создай issue `type:bug` или обнови существующий.
2. **Доказательства.** Собери свежие команды/логи/скриншоты. Один источник не считается достаточным для вывода, если есть второй доступный.
3. **Гипотезы.** Дай 3–5 фальсифицируемых гипотез с ожидаемым наблюдением и командой проверки.
4. **Атрибуция.** Раздели ФАКТ / ГИПОТЕЗА / НЕИЗВЕСТНО, укажи источник каждого факта.
5. **Action gate.** Для мутации напиши узкое меню действий, риск и rollback. Необратимое — через `/greenlight`.
6. **Variant analysis.** После фикса проверь, где ещё может жить тот же класс дефекта.
7. **Handoff.** Добавь в issue блок `AI-CONTEXT`: симптом, доказательства, исключённые гипотезы, следующий безопасный шаг.

## Правило эскалации

Три неудачных фикса подряд — это не «ещё один фикс», а вопрос к архитектуре. Останови попытки, создай диагностический issue или вызови консилиум.

## Артефакты

- Issue инцидента с фазами и `AI-CONTEXT`.
- При необходимости файл `company/incidents/YYYY-MM-DD-<slug>.md`.
- Closing-comment только после проверки восстановления.

## DoD

- [ ] Есть свежий pass/fail-сигнал.
- [ ] Каждая гипотеза проверяема.
- [ ] Любая мутация прошла gate или явно не требовала его.
