# Writing-specs: спека, снимающая мышление

Спека — единственный документ задачи. Исполнитель должен сделать работу, **не открыв ни одного файла «на разведку» и не задав ни одного вопроса**. Не прошло этот тест — спека не готова, диспатч запрещён.

## Структура (носитель: тело issue — `gh issue edit N --body` — или файл `.office/tasks/<slug>.md`; переписывается целиком)

1. **Metadata** — `type`, `dept`, `complexity: S|M|L`, `parallel_safe`, `conflict_paths`, `depends_on`, `executor_profile`.
2. **Цель** — одно предложение: что видит пользователь после мержа. Исходная формулировка задачи растворяется здесь.
3. **Контекст** — файлы и строки, контракты, ловушки (дубли, генерённое). Источник — `graphify query` и отчёты скаутов, не память. Для weak-исполнителей — вырезки кода прямо сюда (см. дисциплину weak-dispatch).
4. **Влияние** — кто зависит от изменяемого (`graphify query "what depends on X"`), что может сломаться. Без этой секции правка не диспатчится.
5. **Решённые развилки** — каждая: решение + обоснование одной строкой. Развилки решает автор спеки, не исполнитель. Меняет скоуп/деньги — стоп, вопрос фаундеру (архитектурная — консилиум).
6. **Инструмент** — ответ: есть ли повторяемое ядро? `нет` / `да, вынос в этой задаче` / `да, отдельный issue type:tooling #N`. Инструмент принадлежит отделу, не персоне. CLI-инструменты должны быть encoding-safe: ASCII для синтаксиса и machine-readable вывода, если кодировка не проверяется отдельно.
7. **Шаги** — нумерованный план правок по файлам; для кода шаг 1 — тест, фиксирующий требование (red → green). Для ambiguous-дизайна сначала tracer bullet / spike.
8. **Границы** — что НЕ делать: не рефакторить попутно, не трогать генерённое, не выходить за ownership отдела.
9. **DoD + проверка** — чек-лист приёмки + конкретная команда/сценарий. Каждое заявление «готово» требует свежего вывода команды или наблюдаемого evidence. Меняется поведение → пункт «обновить доки»; кодовая задача → пункт «graphify update . прогнан».

## Чек-лист выхода

- [ ] Тест самодостаточности: можно исполнить, не открыв ничего сверх спеки?
- [ ] Все развилки решены и записаны (ни одного «на усмотрение исполнителя»)?
- [ ] DoD исполним одной командой/сценарием (verifier не должен ничего изобретать)?
- [ ] Дифф-бюджет реалистичен (для weak: один файл, ≤40 строк — иначе дробить)?
- [ ] Complexity-gate и conflict metadata заполнены?
- [ ] На вопрос «есть ли повторяемое ядро?» дан явный ответ?
- [ ] В DoD требуется свежее доказательство результата, а не «проверил глазами»?
- [ ] Статус выставлен: лейбл `status:spec-ready` (issue) или `status: spec-ready` во frontmatter (файл)?

Симптом на будущее: исполнитель задал вопрос или полез исследовать → это дефект СПЕКИ, не исполнителя. На ретро — правка этой дисциплины.
