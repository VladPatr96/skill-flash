# office

Суперскилл для Claude Code и Codex CLI: **операционная система компании в соло**. `/office` работает в двух треках: лёгкая автоматизация повторяемой рутины без `company/` и полный bootstrap «компании агентов» в любом репозитории. Полный трек ведёт бизнес как конвейер — GitHub Issues со спеками, отделы под функции бизнеса (по готовым blueprints), knowledge-граф, планёрки и отчёты, OKR и unit-экономика, жёсткая токен-экономика. Разовые задачи решает `/solve` — без постоянного процесса.

## Входы

- **`/solve <задача>`** — разовая задача любого домена в любой папке: triage S/M/L, файл-спека `.office/tasks/`, конвейер исполнитель → верификатор, без bootstrap и без GitHub.
- **`/office <цель>`** — постоянный процесс. Intake выбирает трек: «рутина» (мини-состав + инструмент + проверка, без `company/`) или «компания» (полный bootstrap).
- **`/office`** — статус текущего конвейера одной таблицей: done / queued / in-progress / verify / blocked + решения фаундера.

## Командный словарь

| Команда | Что делает |
|---|---|
| `/memo <задача>` | Служебная записка: сырая задача → спека-issue с complexity, conflict metadata и ответом про инструмент |
| `/casting` | Кастинг отдела/агента/инструмента: поиск практик → оффер с unit-оценкой → approve |
| `/greenlight <что>` | Гейт внешнего действия: публикация, деплой, письмо, установка, трата денег |
| `/fire-drill <симптом>` | Инцидент-протокол: доказательства → гипотезы → gated action → handoff |
| `/debrief` | Ретро эпика/недели: zero open items, задержки, уроки, process issues |
| `/corporate` | Месячный business review: OKR, FACT/HYPOTHESIS/UNKNOWN, unit-экономика |
| `/planerka` | Батч решений фаундера ≤5 с рекомендациями |

## Что делает /office

Одна команда `/office <цель>`:

1. **Intake и выбор трека** — короткое grill-me-интервью по нерешённым развилкам (что можно выяснить из репо/веба — выясняет сам). Рутинная цель получает мини-состав scout → worker → verifier и инструмент как часть результата; долгосрочная цель идёт в полный bootstrap.
2. **Bootstrap** — пустой репозиторий превращается в рабочий: git + GitHub remote, лейблы, `company/` (чартеры отделов, ownership-карта, OKR), агенты ядра в `.claude/agents/`, CLAUDE.md с регламентом. Плюс **штатное интервью по blueprint**: готовые оргпроекты по типу бизнеса (`templates/blueprints/` — media, saas, creator, agency, generic) с отделами, промт-паками, OKR-заготовками и метрик-гейтами; профильные отделы предлагаются офферами сразу с unit-оценкой.
3. **Граф знаний** — [graphify](https://github.com/safishamsi/graphify): агенты берут контекст из `graphify query`, а не чтением файлов (главная экономия токенов).
4. **Конвейер** — issue-first: спека в теле issue → диспатч указателем → верификатор со свежим контекстом → приёмка. Оркестратор не пишет код и не читает базу сам.
5. **Найм и инструменты** — отделы и агенты не выдумываются: под задачу ищется практический юзкейс и существующие скиллы (реестр + awesome-списки + поиск), пользователю показывается оффер. Правило v1.0: **инструмент — часть результата**. В каждой спеке есть ответ, есть ли повторяемое ядро; если вынос дешёвый, инструмент создаётся сразу, если дорогой — отдельный `type:tooling`.
6. **Ритуалы** — kickoff/ретро эпиков в `company/meetings/`, недельные и месячные отчёты в `company/reports/`; для high-stakes развилок — консилиум (независимые анонимизированные мнения → синтез).
7. **Инциденты и внешние действия** — `/fire-drill` ведёт диагностику без преждевременных мутаций; `/greenlight` требует проверку, узкое меню и явное подтверждение для публикаций, деплоев, писем, установок и трат.
8. **Усилитель слабых моделей** — `executor-profile: weak`: атомарные спеки (один файл, ≤40 строк диффа) с контекст-пакетом внутри, машинные гейты до верификации, best-of-N в ворктри, tracer bullet, auto-debug после двух реджектов. Дешёвые модели (Haiku-класс) проходят verifier как топовые — регламент выносит «мышление» из модели в процесс.

## Принципы

1. **Простота побеждает** — сначала самое простое и дешёвое работающее решение; доработка отдельными issue по фактам использования.
2. **Инструмент — часть результата** — повторяемое ядро выносится в скрипт/скилл сразу, если это стоит ≤20% задачи; иначе заводится `type:tooling`. Правило трёх остаётся страховкой.
3. **Код — маленькими кусками, тесты — до кода** — модули ≤1000 строк с явными связями, оценка влияния перед правкой, только точечные правки, горячие части выносятся, комментарии на русском, red → green.
4. **Документация всегда актуальна** — README, цели и `company/state.md` обновляются в момент изменения; новая сессия начинает с чтения состояния, а не с раскопок.
5. **Установки — только с согласия** — глобальные тулы (uv, graphify), скиллы, плагины и MCP-серверы ставятся после оффера (что, зачем, команда, что при отказе) и подтверждения пользователя; отказ — фолбэк без переспрашиваний.
6. **Время фаундера — бюджет №1** — неблокирующие вопросы копятся и решаются пакетом на планёрке (≤5 решений с рекомендациями); блокирующие — сразу.
7. **OKR-слой** — цели в `company/okr.md`, эпики привязаны к Key Results, отчёты меряют KR, а не «сделано задач».
8. **Наружу — только с одобрения** — публикации, письма, траты денег агенты не делают сами, пока фаундер явно не делегировал конкретный канал.
9. **Unit-экономика и прозрачность** — каждый оффер несёт «стоимость vs ценность», отчёты сверяют факт с оценкой и объясняют «почему так решили».

Регламент профессионально-нейтрален: конвейер одинаково ведёт фичи программиста и макеты дизайнера (спека = бриф, DoD = команда, сценарий или визуальная приёмка).

## Установка

Как плагин (рекомендуется — обновления через `/plugin`):

```
/plugin marketplace add VladPatr96/office
/plugin install office@office
```

Либо вручную, копией в пользовательские скиллы (удобно для разработки самого скилла):

```powershell
git clone https://github.com/VladPatr96/office
cd office
Get-ChildItem skills -Directory | ForEach-Object { Copy-Item -Recurse -Force $_.FullName "$env:USERPROFILE/.claude/skills/$($_.Name)" }
```

Ставьте одним способом, не обоими сразу — иначе скилл задвоится. После установки доступен во всех проектах как `/office`.

Опциональные зависимости: [gh CLI](https://cli.github.com) (обязателен для конвейера), `uv` + graphify (`uv tool install graphifyy`), плагин Codex (`/plugin marketplace add openai/codex-plugin-cc`).

## Структура репозитория

| Путь | Что это |
|------|---------|
| `.claude-plugin/` | Манифесты плагина и маркетплейса (Claude Code); `.codex-plugin/` — то же для Codex CLI |
| `skills/office/SKILL.md` | Регламент-роутер и единый `/office`-вход |
| `skills/memo`, `skills/casting`, `skills/fire-drill`, `skills/greenlight`, `skills/debrief`, `skills/corporate`, `skills/planerka`, `skills/solve` | **Кирпичи-команды** — маленькие скиллы «одна задача» |
| `skills/weak-dispatch`, `skills/writing-specs`, `skills/verifying` | **Кирпичи-дисциплины** — правила, которые модель включает сама по триггеру |
| `skills/office/templates/` | Шаблоны: спека issue, чартер отдела, карта компании, живой статус (state.md), OKR, планёрка, консилиум, ритуалы (weekly + месячный business review), промт-пак ядра (слот-формат Anthropic prompt library), агенты, блок для CLAUDE.md с правилами кода |
| `skills/office/templates/blueprints/` | Готовые оргпроекты по типу бизнеса: media, saas, creator, agency, generic — отделы, промт-паки, OKR-заготовки, гейты этапов, первые задачи |
| `skills/office/registry.md` | Реестр проверенных скиллов для «найма» + как искать новые |

| Роль | Модель |

| Оркестратор (CEO) | сессионная (Fable / Opus) |

## Происхождение паттернов

Office v1.0 адаптирует идеи из открытой экосистемы, но не копирует чужие файлы. Использованные паттерны: маленькие composable skills и CSO-description (mattpocock/skills), verification-before-completion и systematic debugging (obra/superpowers), issue/state-machine и implementation notes (ccpm/cc-sdd/task-master-подходы), FACT/HYPOTHESIS/UNKNOWN и unit-экономика из product/ops-практик personal-corp-skills. Лицензионная граница: берём механики и пересказываем своими словами; спорные источники с нестандартной лицензией используются только как обзор рынка.
